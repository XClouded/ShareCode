android插件实现与使用

一、传统插件描述

插件的本质在于不修改程序主体（平台）的情况下对软件功能进行扩展与加强，当插件的接口公开后，任何公司或个人都可以制作自己的插件来解决一些操作上的不便或增加新的功能，也就是实现真正意义上的“即插即用”软件开发。平台+插件软件结构是将一个待开发的目标软件分为两部分，一部分为程序的主体或主框架，可定义为平台，另一部分为功能扩展或补充模块，可定义为插件。

插件式架构设计的优点

  • 系统模块独立开发、维护、升级，将繁琐的项目管理工作变得相对方便、灵活；
  • 降低代码的耦合性，降低了系统的维护难度，提高了系统的可扩展性
  • 定义通用接口，将其它功能开放给第三方实现、维护
  • 根据需求动态的组装、分离系统，用户根据自己的需求扩展功能。

二、android插件机制

由于linux平台的安全机制，再加上dalvik的特殊机制，各种权限壁垒，使得开发一个灵活多变的程序，变得比较困难，不像pc平台下那么容易。目前对android插件的接入机制来说主要有三种apk安装，apk不安装，dex包不安装。三种方式其实主要是解决两个方面的问题：１，加载插件中的类，２，加载插件中的资源．第一个加载类的问题，这三个方式都可以很好的解决．但目前三种方式都没有很完美的解决第２个问题。下面分别总结下这三种方式网络上和自己实践出来的一些结论：

1、apk不安装

因为插件没有安装，运行的时候也就没有自己的上下文空间，需要借用主app的上下文空间才能运行，其它和context任何相关的东西都不能使用，也不能获取到drawable、layout下面的资源（传说网络上有牛人有了初步解决方案，但是我没找到），因此界面需要全代码手写而且界面的跳转和service的实现等等更是困难，代码调试起来也比较困难等问题。因为apk没有安装自然不能通过方案一那样设置action来启动activity，该方案的实现逻辑是将SD卡上的.apk文件通过dexclassloader加载到插件中的类，再通过反射的反射的方式调用其中的方法,因此手机并没有给插件创建单独的运行环境，只不过是仅仅是app调用了插件中的方法而已。

（1. ）插件类部分代码

public class MainActivity{
   private Activity hostActivity = null;
   private TextView tvMatch = null;


   protected void createPluginView(Activity activity, Bundle savedInstanceState) {
       this.hostActivity = activity;
       String hostStr = savedInstanceState.getString("function");
       tvMatch = new TextView(hostActivity);
       tvMatch.setText(hostStr);
       hostActivity.setContentView(hostStr);
   }
}

其实就是普通的类，不需要继承activity，也没有activity的生命周期，他完成的主要工作就是创建了一个新的view替换了主Activity的view；因为没有上下文所有不能直接通过R.layout.xxx、R.id.xxx等方式引用布局文件，界面部分的代码也需要手写，也不能直接通过startActivi的方式。另外一个需要注意的地方是插件app中配置manifest文件时activity名称不能省略包名，如不能用android:name=".MainActivity",需要将包名加上写为android:name="com.example.noinstallplugin.MainActivity"。

（2. ）主activity部分代码

 2、apk安装方式

插件apk安装后，插件app也有自己的上下文空间，可以有了插件的context就可以解决加载插件资源的问题．其实这里和我们去调用系统拨号盘、调用浏览器、调用地图app的实现方式和完成的功能都是一样的。不同点就在于这里的插件app不能自启动也不会在程序列表中显示图标，所以对用户来说是不可见的。你可以通过插件app的包名获取到它的图标、名称、版本号等信息，可以在主app中对这些插件app进行安装、更新、卸载等操作。

（1. ）新建插件项目

android项目Manifest.xml中都有如下一段配置
       <activity
           android:name=".PluginMainActivity"
           android:label="@string/app_name" >
           <intent-filter>
               <action android:name="android.intent.action.MAIN" />
               <category android:name="android.intent.category.LAUNCHER" />
           </intent-filter>
       </activity>

其中android.intent.action.MAIN 决定应用程序最先启动的Activity；android.intent.category.LAUNCHER决定应用程序是否显示在程序列表里，但是无MAIN,有LAUNCHER，程序列表中也不会显示该app，因为android.intent.action.MAIN决定应用程序最先启动的Activity，如果没有Main，则不知启动哪个Activity，故也不会有图标出现，其他细节大家可自行研究。

（2. ）为插件启动activity设置启动事件

既然没有启动器，插件应该怎么启动呢，目前的方法为activity设置action，其他app通过action就可以打开插件app（也就和调用系统拨号盘、调用浏览器的方式一样）；将manifest.xml文件修改为
       <activity
           android:name=".PluginMainActivity"
           android:label="@string/app_name" >
           <intent-filter>
                 <action android:name="com.example.suidplugin.MainActivity" />
           </intent-filter>
       </activity>

（3. ）在宿主Activity中调用插件
   public void onStartPluginClicked(View v){
       Intent intent = new Intent("com.example.suidplugin.MainActivity");
       //这里intent也可以传递其他参数信息
       startActivity(intent);
   }

一个非常简易的带插件的app就完成了，两者之间可以完全看作是两个相互独立的app，也不能直接操作对方的数据。可以在插件app启动的时候通过intent传递一些参数信息，主app可以通过异步通信AIDL的方式为插件app提供数据接口，两者之间共享数据可以通过传统数据共享Share preference和content Provider的方式实现。

第二部分源代码下载地址：http://pan.baidu.com/s/1qWK84EC

出现问题：需要在主app中提前知道插件app的action，这里要获取这些数据如果通过网络接口则减缓了加载速度，当网络出现问题的时候就更糟糕了；你可能会想用本地缓存的方式解决，但是android还有更好的解决方案就是SharedUserId。

3、sharedUserId的引入

配置了shareduserid之后，彼此之间就像访问自己的数据一样访问彼此的数据库和文件，主app可以通过SharedUserdId的值查找插件app，插件app可以在打包的时候添加配置文件供主app使用。可以通过设置android:process参数让插件app和主app运行在同一个进程。但是需要注意的是基于安全考虑，两个package需要有相同的签名，否则没有验证就会造成主app数据被其他恶意程序访问。现在可以在插件app中写一个共享文件，主app通过配置文件访问插件app的启动action或者其他的相关配置参数。

public void onStartPluginClicked(View v) throws Exception {
       PackageManager pkgManager = this.getPackageManager();
       String pkgName = this.getPackageName();
       String sharedUID = "";
       try {
           // 获取当前app的suid
           sharedUID = pkgManager.getPackageInfo(pkgName,    PackageManager.GET_UNINSTALLED_PACKAGES).sharedUserId;
       } catch (NameNotFoundException e) {
           e.printStackTrace();
       }


       List<PackageInfo> allPi = pkgManager.getInstalledPackages(PackageManager.GET_UNINSTALLED_PACKAGES);
       List<PackageInfo> pluginPi = new ArrayList<PackageInfo>();

       for (PackageInfo pkg : allPi) {
           // 查找具有相同suid的包，并且排除自身
           if (!sharedUID.equals(pkg.sharedUserId) || pkgName.equals(pkg.packageName))
               continue;
           pluginPi.add(pkg);
       }

       //当前测试代码默认启动其中一个插件
       Context targetContext = this.createPackageContext(pluginPi.get(0).packageName, Context.CONTEXT_INCLUDE_CODE
               | Context.CONTEXT_IGNORE_SECURITY);
       // 根据context读取插件app Assets目录下的文件数据
       InputStream pluginInput = targetContext.getAssets().open("plugin.txt");
       String action = getStrFromInputSteam(pluginInput);
       Intent intent = new Intent(action);
       startActivity(intent);
   }

这里的代码只是抛砖引玉之用，所有实现方式我都把它最简化了，希望大家能很快明白其中主要的思想。

4. dex / jar文件不安装

这个打包插件代码的方式和java开发中jar包的方式相同．和app不安装的方式一样，同样通过dexclassloader加载到插件中的类，依旧没有context,也无法加载到资源．要使用布局，只能在插件中使用java代码来写布局。只是和app不安装实现插件方式不同，打包成jar并不需要完整的工程和依赖关系，单个类也可以单独导出来。（据说可以插件jar文件之间可以共用更加底层的共用jar，但是我始终没找到相应的方法， 如果能实现这个功能那用起来就更加方便了），只是这里到出的jar文件是java bytecode格式的，而android能够直接读取的是Dalvik bytecode格式，所以还需要用DX工具进行一次转化才行。具体方式请点击查看：android导出jar文件并转换问dex格式。打开jar文件可以看到里面有一个calsses.dex的文件，使用.jar或者.dex两者没有任何区别。

三、android插件使用分析

1、目前市场上有那些app运用了插件功能

  • 微信以前的火爆一时的打飞机游戏，当时看过是通过jar插件包的方式实现的，但是今天翻了翻百度居然一片文章也没找到。
  • 有信->发现模块->点我免费送话费和游戏精选两个功能  运用的是下载安装的方式
  • 银联手机支付->我的应用 里面提供了上百种插件安装 使用的是下载安装的方式

带插件的app不多，而且使用了插件你也不一定能发现，因此找这个真有点大海捞针，暂时就只发现这三个了...

2、android插件式app的优点

  • 系统模块独立开发、维护、升级，将繁琐的项目管理工作变得相对方便、灵活；
  • 降低代码的耦合性，降低了系统的维护难度，提高了系统的可扩展性
  • 定义通用接口，将其它功能开放给第三方实现、维护
  • 减小应用安装包的体积
  • 分离出的插件模块可以单独更新、升级。
  • 根据需求动态的组装、分离系统，用户根据自己的需求扩展功能。

3、android插件式app的适用范围

  • 如果你希望在app中集成一些和核心功能不相关的其他功能的时候，这些集成的功能又不是用户的强需求就可以采用这种插件式的设计。
  • 宿主app本身是一个工具箱类型的app，如flyme钱包目前的功能或者更进一步做成生活服务类的app（打车、地铁、购物、团购、外卖、公交、酒店、水电、电影），但是并不是每个用户都需要所有的功能，而且各个模块的更新非常不同步。此时如果做成插件式app，对用户和产品团队而言都是一个比较好的设计。
  • 如果仅仅是为了减小安装包的体积把核心功能一分为二显然是不明智的。同时如果附加功能只有20KB左右，而且整个app就这么一个附加功能显然也没必要做成插件式，其中的权衡需要自己去把握。
